# Инструкции для Claude в этом проекте

## Что это такое

Disposable-прототипы для согласования продуктовых решений внутри команды Documentolog. **НЕ продакшен.**

Цель кода — донести идею за минуту просмотра. После одобрения прототип не дорабатывается, а реализуется заново production-командой по другим стандартам.

## Стек

- Vite + React 19 + TypeScript
- Tailwind CSS v4 (конфиг — в `src/index.css`)
- shadcn/ui (New York style, нейтральная база)
- react-router-dom для роутинга между прототипами
- lucide-react для иконок
- Алиас `@/` → `src/`

UI-тексты — **на русском** (B2B-документооборот, аудитория русскоязычная).

## Главные принципы (по убыванию важности)

1. **shadcn первый, кастом — последний.**
   Прежде чем писать новый компонент, проверь `src/components/ui/`. Если нужного нет — `npx shadcn@latest add <name>`. Список доступных: https://ui.shadcn.com/docs/components.
   Не выдумывай свои Button/Card/Modal.

2. **Prototype-grade код.** Запрещено без явной просьбы:
   - тесты (unit/e2e)
   - error boundaries, try/catch на каждый чих, loading skeletons (если только сам флоу не про загрузку)
   - state-management библиотеки (Redux, Zustand) — `useState` + props хватит
   - абстракции «на вырост» (generic-хуки, фабрики компонентов)
   - react-hook-form + zod для 1–2 полей — обычный `useState` достаточно
   - i18n, backend, fetch/axios, аутентификация
   - code-splitting, dynamic import, оптимизации бандла
   - иконки сверх Material Symbols и lucide (только в shadcn UI компонентах)

   Разрешено и поощряется:
   - хардкод данных прямо в файле экрана
   - копипаст между прототипами (изоляция > DRY)
   - inline-стили в виде tailwind-классов
   - `any` в моках если он экономит время

3. **Лимит сложности.** Если флоу разрастается до 5+ экранов или появляется реальная бизнес-логика (валидации, расчёты, состояния которые надо синхронизировать) — **СТОП**. Спроси:
   > «это всё ещё прототип или мы строим MVP?»
   Прототип должен читаться целиком за один присест.

4. **Mock-данные:**
   - общие сущности (юзеры, орг-структура, типовые документы) — в `src/shared/mock/`
   - специфика флоу — в `src/prototypes/{slug}/data.ts`
   - никаких API-вызовов

## Как добавить новый прототип

1. Создать папку `src/prototypes/YYYY-MM-{kebab-case-тема}/`
2. Внутри:
   - `index.tsx` — default-экспорт React-компонента. Обычно обёрнут в `<AppShell title="...">`. Если экранов несколько — используй локальные `<Routes>` от react-router-dom (внешний путь `/p/:slug/*` уже поддерживает их).
   - `screens/` — отдельные экраны (если их 2+)
   - `data.ts` — локальные моки
   - `README.md` — 5 строк: **гипотеза**, **что показываем**, **открытые вопросы**, **аудитория демо**
3. Зарегистрировать в `src/prototypes/registry.ts`:
   - импортнуть default-экспорт прототипа
   - добавить запись в массив `prototypes` **в начало** (чтобы был сверху галереи)
4. Открыть `http://localhost:5173/` и проверить что карточка появилась
5. Перейти в прототип — в шапке кнопка «Все прототипы» возвращает в галерею

## Стиль кода

- TypeScript везде, `any` допустим в моках
- Импорты через алиас: `@/components/ui/button`, `@/shared/layout/AppShell`, `@/shared/mock/users`
- Иконки в прототипах — **только Material Symbols** через `<Icon name="..." />` из `@/shared/primitives/Icon`. Имена на [fonts.google.com/icons](https://fonts.google.com/icons). Исключение: `lucide-react` допустим внутри shadcn UI компонентов (`src/components/ui/*`) — их не трогать.
- Цвета и радиусы — только через CSS-переменные shadcn (`bg-primary`, `text-muted-foreground`, `rounded-lg`). Не хардкодить hex.
- `src/components/ui/*` — **только через `shadcn add`**, руками не править

## Брендинг

Точка кастомизации — `src/index.css`. Чтобы приблизить к корпоративной ДС: подменить значения `--primary`, `--radius`, при необходимости `--accent`. Значения в формате `oklch()`. Темная тема настраивается в селекторе `.dark` отдельно.

## Шаринг

- **Локально:** `npm run dev` → `http://localhost:5173/dg-design-prototyping/`
- **Онлайн (для команды):** GitHub Pages, авто-деплой при каждом push в `main`. См. раздел «Git, перенос и онлайн-деплой» выше.

## Git, перенос и онлайн-деплой

Репозиторий: **публичный GitHub-репо `dg-design-prototyping`**, онлайн-версия — **GitHub Pages с авто-деплоем через GitHub Actions** (`.github/workflows/deploy.yml`). Каждый push в `main` пересобирает сайт и публикует все прототипы.

**Онлайн-адрес:** `https://<github-username>.github.io/dg-design-prototyping/`
Конкретный URL смотри в выводе `gh repo view --web` или в Settings → Pages репозитория.

### Запушить (по фразе или команде)
Когда пользователь говорит **«отправь в гитхаб»**, **«запушь»**, **«push»**, **«сохрани в гит»** или вызывает `/push` — выполни полный цикл без доп. подтверждения (намерение уже выражено):
1. `git add .`
2. `git commit` — короткое осмысленное сообщение на основе diff
3. `git push`

После push GitHub Actions сам пересоберёт и опубликует онлайн-версию за ~1–2 мин. Статус прогона: `gh run list` / `gh run watch`.

### Перенос на новый комп или коллеге
```
gh auth login          # разово на новой машине
git clone https://github.com/<username>/dg-design-prototyping
cd dg-design-prototyping && npm install && npm run dev
```

### ⚠️ Связка base ↔ имя репо
В `vite.config.ts` стоит `base: '/dg-design-prototyping/'` — он **обязан совпадать с именем репозитория**, иначе онлайн-сборка ляжет белым экраном (бандл не найдёт JS/CSS). При переименовании репо меняй `base` синхронно.

Роутинг — **HashRouter** (`src/main.tsx`), URL вида `/#/p/slug`. Это нужно, чтобы deep-link и F5 работали на статике Pages. Локально (`npm run dev`) сайт живёт на `http://localhost:5173/dg-design-prototyping/`.

## Когда сомневаешься

Выбирай скорость показа над архитектурной чистотой. Прототип, который написан за 30 минут и показывает идею — лучше, чем прототип за 3 часа с красивой структурой.
